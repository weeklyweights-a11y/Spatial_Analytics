# Phase 3: Scoring Engine + CCTV Wall

## What This Phase Does

Make the system think and show. The scoring engine consumes activity events from Redis Stream, calculates weighted scores for all 1,000 participants, assigns behavioral tags, and writes results to PostgreSQL and Redis. The CCTV wall dashboard goes live — the organizer opens a browser and sees a grid of all camera feeds with every person labeled, color-coded by activity. Click on any person in the video and a score card pops up with their name, rank, radar chart, and tags. WebSocket channels push live updates so the dashboard stays current without refreshing.

This is the MVP. If Phase 3 works, you have a shippable product for Buildathon Dallas. Everything after this phase is enhancement.

## Prerequisites

- Phase 2 complete and all acceptance criteria passing
- Camera workers emitting activity events to Redis Stream
- Annotated frames being generated by camera workers (bounding boxes + names)
- At least 2 test camera streams running via simulate_streams.py
- At least 5 test participants registered with faces that appear in the test videos

## Section 1: Scoring Worker

### What to build

`backend/workers/scoring_worker.py` — a standalone Python process that consumes activity events from Redis Stream, aggregates them, and calculates scores.

**Process lifecycle:**
1. Connect to Redis and PostgreSQL
2. Load scoring weights from scoring_config table
3. Enter main loop: consume → aggregate → score → persist → push

**Main loop (every 60 seconds):**

Step 1 — Consume events from Redis Stream:
- `XREAD` from `activity_stream` since last consumed ID
- Collect all events into a buffer
- Track the last consumed event ID for the next cycle

Step 2 — Aggregate by participant:
- Group events by `participant_id`
- For each participant, count events per activity type
- Convert event count to minutes: `minutes = event_count * (flush_interval / total_events_for_participant)`
- This handles the case where a participant switches activities mid-cycle

Step 3 — Apply scoring weights:
- For each activity, check `min_dwell_seconds` from scoring_config. If the participant spent less than the minimum in that activity during this cycle, don't score it (prevents drive-by scoring from walking through a zone)
- Calculate: `period_points = Σ (activity_minutes × activity_weight)`

Step 4 — Update scores in PostgreSQL:
- For each participant, UPDATE the scores table:
  - `total_score += period_points`
  - `coding_minutes += this_cycle_coding_minutes` (and same for all activity types)
  - `last_zone = most recent zone from events`
  - `last_activity = most recent activity from events`
  - `last_seen_at = most recent timestamp`
  - `updated_at = NOW()`
- Use batch UPDATE (not one query per participant) for performance

Step 5 — Calculate behavioral tags:
- After updating minutes, recalculate tags for each participant who was active this cycle:
  - `total_minutes = sum of all activity minutes`
  - Builder: `coding_minutes / total_minutes > 0.50`
  - Mentor: `(mentoring_minutes + helping_minutes) / total_minutes > 0.15`
  - Collaborator: `collaborating_minutes / total_minutes > 0.30`
  - Networker: `networking_minutes / total_minutes > 0.20`
  - Night Owl: `last_seen_at.hour >= 2 AND last_seen_at.hour < 5`
  - Cross-Pollinator: participant detected in 3+ different team zones (requires tracking which zones they've visited — store in a Redis set `visited_zones:{participant_id}`)
- UPDATE tags array in scores table

Step 6 — Update Redis:
- `ZADD leaderboard score participant_id` for every active participant
- Recalculate ranks: `ZREVRANK leaderboard participant_id` for each
- UPDATE rank in PostgreSQL scores table

Step 7 — Write activity logs to PostgreSQL:
- Batch INSERT all consumed events into activity_logs table
- This is the permanent record of every detected activity

Step 8 — Push updates via WebSocket:
- Publish a "scores_updated" message to a Redis Pub/Sub channel
- The API server subscribes to this channel and pushes to connected WebSocket clients (see Section 3)

**Flush cycle timing:** The scoring worker runs its loop every `SCORING_FLUSH_INTERVAL` seconds (default 60). Each loop should complete in <5 seconds for 1,000 participants. If it takes longer, log a warning with the actual duration.

**Startup recovery:** On startup, check if there are unprocessed events in Redis Stream (events added while the worker was down). Process them immediately before entering the regular loop. Use a persistent last-consumed-ID stored in Redis (`SET scoring_last_id {id}`).

**Graceful shutdown:** On SIGTERM, complete the current flush cycle, persist the last consumed event ID, exit.

## Section 3: WebSocket Channels

### What to build

`backend/api/websocket.py` — WebSocket endpoints that push live data to connected dashboards.

**Authentication:** WebSocket connections authenticate via JWT token in query parameter: `ws://host/ws/leaderboard?token=xxx`. On connection, validate the token. If invalid, close with 4001 code. If the token expires while connected, close with 4002 code.

**Channel: /ws/leaderboard**
- Pushes every 30 seconds
- Payload:
  ```json
  {
    "type": "leaderboard",
    "data": [
      {
        "participant_id": "uuid",
        "name": "Arjun Patel",
        "team_name": "Team Beta",
        "total_score": 1204.5,
        "rank": 1,
        "current_activity": "coding",
        "current_zone": "Coding Zone A",
        "tags": ["Builder", "Night Owl"]
      }
    ],
    "total_participants": 1000,
    "timestamp": "2026-07-15T04:30:00Z"
  }
  ```
- Source: Redis leaderboard (ZREVRANGE with scores) + participant state hashes

**Channel: /ws/tracking/{camera_id}**
- Pushes every 1 second (tied to camera worker frame rate)
- Payload:
  ```json
  {
    "type": "tracking",
    "camera_id": "CAM-01",
    "persons": [
      {
        "track_id": 42,
        "participant_id": "uuid",
        "name": "Riya Sharma",
        "bbox": [120, 80, 280, 350],
        "activity": "coding",
        "confidence": 0.94
      }
    ],
    "timestamp": "2026-07-15T04:30:01Z"
  }
  ```
- Source: camera worker publishes tracking data to Redis Pub/Sub channel `tracking:{camera_id}`. The WebSocket handler subscribes to this channel and forwards to connected clients.
- This channel powers the click-to-inspect feature: the frontend knows where each person's bounding box is and can match click coordinates to a participant_id.

**Channel: /ws/participant/{participant_id}**
- Pushes every 30 seconds (when the participant's score changes)
- Payload:
  ```json
  {
    "type": "participant_update",
    "participant_id": "uuid",
    "name": "Riya Sharma",
    "zone": "Coding Zone A",
    "activity": "collaborating",
    "score": 918.0,
    "rank": 7,
    "tags": ["Builder", "Mentor"],
    "timestamp": "2026-07-15T04:30:00Z"
  }
  ```
- Source: Redis participant state hash. Pushed when scoring engine signals an update.

**Connection management:**
- Track all active WebSocket connections in a set per channel
- On disconnect, remove from set
- Log connection count periodically: "WebSocket connections: leaderboard=3, tracking=5, participant=1"
- Max connections per channel: 50 (rate limit to prevent resource exhaustion)

**How the scoring worker triggers WebSocket pushes:**
- After each flush cycle, the scoring worker publishes to Redis Pub/Sub channel `scores_updated`
- The API server's WebSocket manager subscribes to `scores_updated`
- On receiving the signal, it reads fresh data from Redis and pushes to all connected leaderboard and participant WebSocket clients
- For tracking channels: camera workers publish directly to Redis Pub/Sub `tracking:{camera_id}` on every frame. The WebSocket handler forwards these to connected clients.

## Section 4: Annotated Frame Streaming (MJPEG)

### What to build

The mechanism to get annotated video frames (with bounding boxes and names drawn by camera workers in Phase 2) from the camera workers to the browser dashboard.

**Approach: MJPEG over HTTP**

Each camera worker encodes its annotated frame as JPEG after every processing cycle and stores it in a shared location. The API server exposes an MJPEG endpoint per camera that continuously serves the latest frame.

**Camera worker side:**
- After annotating the frame (Phase 2 Section 4, Step 6), encode it as JPEG bytes
- Publish to Redis: `SET camera_frame:{camera_id} {jpeg_bytes}` with 5-second TTL
- This gets overwritten every frame (~10 FPS), so Redis always has the latest frame

**API server side:**
- `GET /api/v1/stream/{camera_id}` — MJPEG endpoint
- Content-Type: `multipart/x-mixed-replace; boundary=frame`
- In a loop: read `camera_frame:{camera_id}` from Redis, yield as MJPEG frame, sleep 100ms (10 FPS)
- This is a streaming HTTP response — the connection stays open and the browser renders it as a live video feed

**Dashboard side:**
- Each camera feed in the CCTV wall grid is an `<img>` tag with `src="/api/v1/stream/CAM-01"`
- The browser handles MJPEG natively — the image continuously updates without JavaScript
- No WebRTC, no HLS, no video.js — just a plain `<img>` tag with MJPEG source

**Performance:** Redis stores ~100KB per frame (JPEG at 720p quality 70). With 13 cameras updating at 10 FPS, that's ~13MB/s of Redis writes. Redis handles this easily. The dashboard browser fetches at its own pace (probably 5-10 FPS effective).

**Important:** The MJPEG stream does NOT have bounding box click coordinates. The click-to-inspect feature uses the /ws/tracking/{camera_id} WebSocket channel which sends bounding box coordinates separately. The dashboard overlays invisible click targets on top of the MJPEG feed based on the WebSocket tracking data.

## Section 5: CCTV Wall Dashboard

### What to build

`dashboard/src/pages/CCTVWall.tsx` — the primary dashboard view. A grid of live camera feeds with click-to-inspect score cards.

**Layout:**
- Full-screen grid of camera feeds, organized by floor
- Top bar: "SpatialScore — CCTV Wall" + user menu + settings gear
- Camera grid takes ~70% of screen width and full height
- Right sidebar (~30%): mini leaderboard (top 10), energy level bar, alerts feed
- Bottom bar: mini heatmap (optional, can be a tab)

**Camera grid:**
- Cameras grouped by floor: "Ground Floor" | "1st Floor" | "2nd Floor" column headers
- Each camera tile shows:
  - Camera name label (e.g., "CAM-01 Coding Zone A")
  - Live MJPEG feed via `<img src="/api/v1/stream/{cam_id}">`
  - Invisible click overlay: a transparent `<div>` positioned over the image that captures click events
  - Person count badge in corner (from zone occupancy data)
- Responsive: 3-4 columns on wide screens, 2 columns on laptops

**Click-to-inspect overlay:**
- The /ws/tracking/{camera_id} WebSocket sends bounding box coordinates for every person every second
- The dashboard renders invisible clickable rectangles positioned over each person in the video
- These rectangles are absolutely positioned `<div>` elements layered on top of the `<img>` tag
- The position is calculated by mapping the bounding box coordinates (from the camera worker's frame resolution) to the displayed image size
- Each rectangle has an onClick handler that opens the score card for that participant_id

**Handling coordinate mapping:**
- Camera worker processes at the native frame resolution (e.g., 1920x1080)
- The MJPEG stream is displayed at a smaller size in the browser (e.g., 480x270 per tile)
- The bounding box coordinates from the WebSocket are in native resolution
- Scale them: `displayed_x = (native_x / native_width) * displayed_width`
- Apply this scaling to all 4 coordinates of the bounding box

**Score card popup:**
- Appears when a person's bounding box is clicked
- Positioned near the clicked location (but adjusted to stay on screen)
- Shows:
  - Registration photo (from /api/v1/participants/{id} response, served as static file)
  - Name, team name, track
  - Current zone + activity (live, via /ws/participant/{id})
  - Total score + rank out of 1,000
  - Mini radar chart (Recharts RadarChart): 5 axes — Coding, Collaborating, Mentoring, Presenting, Networking. Values are percentages of total time.
  - Behavioral tags as colored badges
  - "View Full Profile" link → navigates to /participant/{id}
- Clicking another person's box updates the card
- Clicking empty space or pressing Escape closes the card
- The card subscribes to /ws/participant/{id} for live score updates while open

**Mini leaderboard sidebar:**
- Shows top 10 participants by total score
- Each entry: rank, name, score, current activity icon
- Updates via /ws/leaderboard every 30 seconds
- "View All" link → navigates to /leaderboard

**Alerts feed sidebar:**
- Shows recent alerts as a scrolling list
- Subscribes to /ws/alerts (built in Phase 4, but the UI shell exists now)
- For Phase 3: just show "No alerts" placeholder

**Components to build:**

`CameraFeed.tsx` — single camera tile with MJPEG image and click overlay
- Props: cameraId, cameraName, floor
- Subscribes to /ws/tracking/{cameraId} for bounding box positions
- Renders MJPEG `<img>` + overlay divs for each tracked person
- onClick on a person div → calls onPersonClick(participantId)

`ScoreCard.tsx` — the popup that appears when clicking a person
- Props: participantId, position (x, y on screen)
- Fetches participant data from GET /api/v1/scores/{id}
- Subscribes to /ws/participant/{id} for live updates
- Renders photo, name, score, radar chart, tags
- Recharts RadarChart with 5 axes

`RadarChart.tsx` — reusable radar chart component
- Props: data array of {axis, value} where value is 0-1
- Uses Recharts RadarChart, PolarGrid, PolarAngleAxis

`LiveLeaderboard.tsx` — mini leaderboard for sidebar
- Subscribes to /ws/leaderboard
- Renders ranked list of top N participants

`AlertsFeed.tsx` — scrolling alert list for sidebar
- Subscribes to /ws/alerts (placeholder for Phase 3)
- Renders alert messages with severity icons

## Section 6: Score Endpoints

### What to build

REST endpoints that the dashboard and score card use to fetch detailed participant data.

**GET /api/v1/scores/leaderboard**
- Query params: `?sort_by=total_score|coding|mentoring|collaborating|networking&sort_order=asc|desc&limit=50&page=1&tag=builder|mentor`
- Response: paginated list of participants with scores, ranks, current activity, tags
- Source: JOIN participants + scores tables, ordered by requested sort field
- Cache: first call hits PostgreSQL, result cached in Redis for 30 seconds (invalidated by scoring engine flush)

**GET /api/v1/scores/{participant_id}**
- Response: full score breakdown including:
  - total_score, rank
  - Per-activity minutes (coding, collaborating, mentoring, presenting, networking, helping, idle)
  - Per-activity points (minutes × weight)
  - Per-activity percentage of total time
  - radar_data: array of {axis, value} for the radar chart (percentages)
  - tags array
  - current_zone, current_activity (from Redis participant state)
  - registered_at, last_seen_at
- Source: scores table + Redis participant state

**GET /api/v1/scores/{participant_id}/timeline**
- Response: hour-by-hour breakdown of activities
  ```json
  {
    "timeline": [
      {"hour": "18:00", "zone": "Coding Zone A", "primary_activity": "coding", "minutes": 52},
      {"hour": "19:00", "zone": "Coding Zone A", "primary_activity": "collaborating", "minutes": 45},
      ...
    ]
  }
  ```
- Source: activity_logs table, grouped by hour and primary activity
- Note: this endpoint can be slow for participants with many activity logs. Use the hourly partitions for query efficiency.

**GET /api/v1/tracking/active**
- Response: all currently active participants with their locations
- Source: Redis participant state hashes (all `participant:*` keys where last_seen is within last 5 minutes)
- Used by the CCTV wall to show which participants are currently visible

## Section 9: Error Handling in Real-Time Pipeline

### What to build

Error handling patterns specific to the real-time scoring and WebSocket pipeline.

**Scoring engine errors:**
- If PostgreSQL is temporarily unavailable during a flush: log error, hold events in buffer, retry next cycle. Don't lose events.
- If a single participant's score update fails: log error with participant_id, skip that participant, continue with others. Don't let one bad record block all scoring.
- If Redis Stream read fails: log error, wait 5 seconds, retry. The stream consumer is idempotent (we track last consumed ID).

**WebSocket errors:**
- Client disconnect: remove from connection set, no error logging (normal behavior)
- Client sends invalid message: log warning, ignore message, don't disconnect
- Redis Pub/Sub disconnect: reconnect with exponential backoff, log warning. WebSocket clients will see stale data until reconnected.
- Too many connections: reject new connections with 4003 code if channel is at 50 connections

**MJPEG stream errors:**
- If camera_frame:{camera_id} key doesn't exist in Redis (camera worker is down): serve a static "Camera Offline" placeholder image
- If Redis is temporarily unavailable: serve last known frame (cache in API server memory), log warning
